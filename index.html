What is a (quantifiable) performant animation?

The holy grail: 60 frames per second

  • FPS = how many frames can be rendered in one second
  • A measurement for a site's "responsiveness"
  • 60 FPS feels natural to the human eye
  • More calculations = less frames

Why is this important?

  • Animations change the currently painted state of the UI
  • The browser needs to calculate the changes for each frame
  • Calculations need to be as quick as possible
  • Less calculations -> less time per frame -> more frames per second

So?

  • A CSS animation changes the value of a property over time
  • Different properties trigger different rendering states (https://csstriggers.com)
  • Key to 60 FPS is animating 'cheap' properties

Know your friends

  • Properties that only trigger Composition
    • opacity
    • transform

Every animation should be broken down to using opacity or transform

Tell the browser what will-change

will-change

  • Tells the browser that something is going to change
  • will-change: opacity;
    • The opacity of this element is going to change soon
  • will-change: scroll-position;
    • The scroll position of this element is going to change soon
  • will-change: transform;
    • This element is going to be transformed soon

will-change: caution !important

  • Don't apply will-change to too many elements
  • Apply it only directly before the animation is going to happen
  • ...or to elements that are being animated very often
  • Don't use it if you don't need to â€“ don't fix it if it ain't broken!
  • Also remove it when it is not needed any more
  • Be aware that there are side effects
    • will-change creates a new stacking layer
    • Especially absolute and fixed positioned elements can be affected by that in unexpected ways!
